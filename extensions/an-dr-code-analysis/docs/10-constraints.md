# 10 — Constraints & Non-Goals

## 10.1 Known Limitations

### C / C++

- **Macro-expanded calls**: clangd handles these; ctags and regex do not.
  When using ctags/regex fallback, calls hidden inside macros will be missing.
  Always flag this in the confidence warning when fallback is active.

- **Template instantiations**: clangd resolves these; ctags does not.
  Call graphs for heavily templated C++ code are unreliable without clangd.

- **No build system**: without `compile_commands.json` or a detectable build system,
  clangd cannot resolve cross-file includes accurately.
  Directory heuristic is the best available fallback for component deps.

- **Cross-compilation**: clangd may reject flags intended for embedded toolchains.
  Mitigated by `.clangd` generation (see 04-toolchain.md), but not fully solvable
  without user configuration effort.

- **Generated code**: files generated by code generators (protobuf, Qt moc, etc.)
  may produce confusing graph entries. No filtering is applied automatically.

### Python

- **Dynamic imports**: `importlib.import_module()`, `__import__()`, and plugin
  systems with runtime discovery are invisible to all static analysis tools.
  There is no workaround — document this clearly in the UI.

- **Conditional imports**: `if sys.platform == 'win32': import winreg` — these are
  included in analysis as if always imported. May produce platform-specific noise.

- **pyan3 limitations**: pyan3 performs name-based (not type-based) call analysis.
  It may produce false positives when different functions share the same name.

### Rust

- **Proc macros**: calls generated by procedural macros may not appear in call graphs.
  rust-analyzer handles most common cases but not all.

- **Dynamic dispatch (trait objects)**: `Box<dyn Trait>` call sites cannot be fully
  resolved statically. These edges will be missing from call graphs.

### TypeScript / JavaScript

- **Dynamic requires**: `require(variablePath)` is not resolvable statically.

- **Barrel re-exports**: deep re-export chains (`index.ts` re-exporting from
  multiple files) may produce noisy file dependency graphs.

- **JavaScript accuracy**: JS has no type information; tsserver analysis quality
  for `.js` files is lower than for `.ts` files.

## 10.2 Explicit Non-Goals

The following are **out of scope** for this extension. Do not implement them,
even partially, without explicit requirement change.

- **Documentation generation**: no Doxygen, JSDoc, or rustdoc output
- **Code editing or refactoring**: read-only analysis only
- **Runtime / dynamic analysis**: no execution, no profiling, no tracing
- **AI provider management**: handled entirely by companion extension
- **Remote code execution**: all analysis runs locally
- **Test coverage**: no integration with test frameworks
- **Metrics / complexity analysis**: cyclomatic complexity, line counts, etc.
- **Diff / change tracking**: no git integration
- **Export to external tools**: no Confluence, Notion, or wiki export
  (PNG/SVG export from expanded view is in scope as a future item)
- **Team / shared analysis**: no collaboration features
- **Language server implementation**: this extension is a consumer of LSP, not a server

## 10.3 Doxygen Decision

Doxygen is explicitly excluded as a dependency because:

1. Requires annotated source code (`/** @brief */`) — real codebases rarely have
   complete coverage
2. Requires a full Doxygen build run — slow, config-heavy, not always set up
3. clangd provides equivalent call/dependency data without these prerequisites

**Exception**: if a project already has a `Doxyfile` and pre-generated XML output
(`{buildDir}/xml/`), the extension MAY parse the XML as an optional data source
in a future version. This is deferred, not planned.

## 10.4 Build Order / Implementation Priority

Implement in this order. Each item should be functional before starting the next.

1. Extension scaffolding — activates, registers commands, shows empty sidepanel
2. Tool detection — ToolRegistry detects all tools, TOOLS STATUS section populates
3. Context tracking — CONTEXT section updates on editor changes
4. clangd call graph — LSP integration for C/C++, graph renders in sidebar
5. Cytoscape renderer — basic graph with click/double-click interactions
6. ctags fallback — fallback chain for call graph
7. Confidence badges — every result tagged with tool + confidence level
8. File deps (C/C++) — clangd + #include parse
9. rust-analyzer integration — call graph + file deps for Rust
10. cargo metadata — Rust component deps
11. tsserver integration — TypeScript call graph + file deps
12. tsconfig component deps — TypeScript component graph
13. Python (pyan3 + AST walk)
14. Component deps (CMake + directory heuristic)
15. clangd recovery actions — compile_commands generation, .clangd generation
16. Analysis cache — mtime-based invalidation
17. Expand to full tab — wider graph view
18. AI fallback bridge — ExternalExtensionAnalyzer + confirmation flow
19. Polish — filters, depth controls, node context menu, output channel logging

## 10.5 Versioning Contract for AI Bridge

The inter-extension contract (Section 5.3) must be versioned.
Breaking changes to `AiAnalysisRequest` or `AiAnalysisResponse` require a version bump.

Include a `contractVersion: string` field in `AiAnalysisRequest`.
The companion extension should reject requests with unsupported versions gracefully.

Initial version: `"1.0"`
